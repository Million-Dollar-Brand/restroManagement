# Plan to Fix Missing Staff Table Error

## Problem Analysis
The error `relation "staff" does not exist` occurs when accessing `/api/v1/staff` because the staff table is missing from the tenant schema. The application uses multi-tenant architecture where each restaurant has its own PostgreSQL schema (e.g., `tenant_uuid_replaced`), and tenant-specific tables like `staff` are created in these schemas.

## Root Cause
1. **Tenant Tables Creation**: The code creates tenant tables using `TenantBase.metadata.create_all()` in the `create_tenant_tables()` function when a new restaurant is created.
2. **Existing Restaurants**: If restaurants exist from before this table creation logic was implemented, their schemas may not have the `staff` table.
3. **Migration Branch**: There's a separate Alembic migration branch (`d12345678901_create_tenant_schema_tables.py`) for tenant tables, but it's not integrated into the main migration chain and may not have been applied to existing tenant schemas.

## Solution Plan
Implement a data migration/initialization script to ensure all existing tenant schemas have the required tables.

### Steps:
1. **Create Initialization Script**: Develop a Python script that:
   - Connects to the database
   - Queries all active restaurants from the `public.restaurants` table
   - For each restaurant, switches to its tenant schema
   - Creates the missing tenant tables using `TenantBase.metadata.create_all()`
   - Optionally populates initial data if needed

2. **Handle Schema Context**: Ensure the script properly sets the schema on `TenantBase.metadata` before creating tables, similar to how `create_tenant_tables()` does it.

3. **Error Handling**: Include proper error handling for cases where schemas don't exist or creation fails.

4. **Run the Script**: Execute the script in a safe environment (development/staging first) to create missing tables.

5. **Verification**: After running, test the `/api/v1/staff` endpoint to confirm the table exists and queries work.

6. **Future Prevention**: Consider adding a check in the tenant session dependency to validate table existence and create tables on-the-fly if missing (though this may impact performance).

## Implementation Details
- Location: Create `scripts/init_tenant_tables.py`
- Dependencies: Use existing database connection logic from `app.database`
- Models: Import `TenantBase` from `app.models.tenant`
- Safety: Include dry-run mode and confirmation prompts

## Risks
- **Data Loss**: Low risk since `create_all()` only creates missing tables
- **Performance**: Script may take time if many restaurants exist
- **Downtime**: Can be run without taking the app offline if using a separate connection

## Testing
- Test on a copy of production data first
- Verify table creation in multiple tenant schemas
